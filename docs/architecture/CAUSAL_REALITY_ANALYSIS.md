## Executive Summary

The Khepra Protocol positions itself philosophically opposite to "compliance theater" by implementing **causal risk attestation** grounded in cryptographic proofs, post-quantum security, and directed acyclic graphs (DAGs) of trust relationships. This analysis evaluates the project against the "roots vs leaves" framework—assessing whether the implementation prioritizes deep foundational capabilities over superficial compliance features.

**Verdict**: Khepra is a **roots-first framework** with disciplined architectural boundaries. The project implements production-grade security primitives (post-quantum crypto, causal modeling, immutable attestation) while explicitly avoiding feature bloat.

Aligned with the NouchiX "Causal Defense Intelligence" framework, Khepra embodies:

### 1.1 Chatham House Rule: Share Insights, Protect Identities
**Implementation**:
- Intelligence fusion from CISA KEV, MITRE ATT&CK, Shodan, Censys
- No vendor lock-in or proprietary threat intel dependencies

**Implementation**:
- Active scanning via Sonar agent (port enumeration, banner grabbing)
- OSINT correlation to identify public exposure (Shodan/Censys integration)
- Adversarial simulation capabilities through Arsenal tool integration (ZAP, RetireJS, Gitleaks)

**Evidence**: `cmd/sonar/main.go` and `pkg/sonar/orchestrator.go` implement active reconnaissance.

### 1.4 No Vendor Pitches: This is Peer Learning, Not Prospecting

- Tool-agnostic architecture (normalizes external scanner outputs)

## 2. Roots vs Leaves: Architectural Assessment

### 2.1 THE ROOTS (Deep Foundational Capabilities)

#### Root 1: Post-Quantum Cryptography
**Files**: `pkg/adinkra/hybrid_crypto.go`, `pkg/adinkra/adinkra_core.go`

**Implementation Depth**:
- **NIST ML-DSA-65 (Dilithium3)**: 3,293-byte signatures for non-repudiable attestations
- **NIST ML-KEM-1024 (Kyber)**: 1,568-byte ciphertexts for key encapsulation
- **Triple-layer hybrid encryption**: Khepra-PQC + Dilithium signing + ECDSA fallback
- **Memory hygiene**: Explicit secret wiping after cryptographic operations

**Why This is a Root**:
- Addresses NIST 2026+ mandates for quantum-safe migration
- Not aspirational—actively integrated into every attestation and encryption operation
- Prevents "store-now-decrypt-later" attacks by nation-state adversaries

**Evidence of Maturity**:
```go

```

#### Root 2: Causal Risk Graphs (DAG-Based Attestation)
**Files**: `pkg/dag/dag.go`, `pkg/attest/attest.go`, `pkg/intel/risk_engine.go`

**Implementation Depth**:
- **Content-addressed immutable DAG**: Prevents retroactive evidence tampering
- **Cryptographically signed nodes**: Every attestation includes Dilithium3 signature
- **Causality chains**: Risks are not isolated findings—they're linked to root causes

**Why This is a Root**:
- Moves from "checklist compliance" to "provable security posture"
- Enables forensic analysis: "If SSH is compromised, what dependent systems fail?"
- Client-verifiable proofs (no trust required in auditor claims)

**Evidence of Maturity**:

#### Root 3: Distributed Consensus & Continuity (DRBC)
**Files**: `pkg/drbc/genesis.go`, `pkg/scorpion/scorpion.go`

**Implementation Depth**:
- **Genesis Protocol**: AES-256-CTR streaming encryption for infrastructure snapshots

- **Argon2 key stretching**: Prevents brute-force decryption

**Why This is a Root**:
- Ensures operational continuity with cryptographic binding
- Not just backups—immutable, timestamped, sealed integrity proofs
- Prevents "shadow IT" restoration (unauthorized decryption attempts locked out)

**Evidence of Maturity**:

**Files**: `pkg/intel/loader.go`, `pkg/intel/mitre.go`, `pkg/sonar/shodan.go`

**Why This is a Root**:

### 2.2 THE LEAVES (Surface Compliance & Acknowledged Peripherals)

#### Leaf 1: LLM Integration (Optional Cognitive Layer)
**Files**: `pkg/agi/engine.go`

**Implementation**:
- Generates narrative risk summaries (not primary analysis)
- Falls back to heuristics when LLM unavailable
- Uses Ollama (local models) for privacy

**Why This is a Leaf**:
- Not foundational to security posture
- Enhances UX but doesn't change cryptographic proofs
- Explicitly optional (can disable without losing core functionality)

#### Leaf 2: External Scanner Integration (Arsenal)
**Files**: `pkg/arsenal/` (Gitleaks, TruffleHog, ZAP, RetireJS)

**Implementation**:
- Normalizes outputs from third-party tools into DAG nodes
- Khepra doesn't replace scanners—it contextualizes their findings

**Why This is a Leaf**:
- Integration layer, not foundational capability
- Adds breadth (more vulnerability types) but not depth (risk causality)
- Tools are swappable (architecture is tool-agnostic)

#### Leaf 3: Application-Level STIG Compliance
**Files**: Not implemented natively

**Acknowledged Gap**:
- OS-level compliance enforced natively
- Application-level requires external input (Nessus, Kubebench)

**Rationale for Omission**:
- Prevents scope creep (hundreds of app-specific STIGs)
- Focuses on foundational OS security (maximum impact)

## 3. Critical Gaps: Roots Being Skipped

### Gap 1: Runtime Integrity Monitoring (FIM)
**Current State**: File Integrity Monitoring mentioned in Tier 2 monetization but not implemented

**Why This is a Missing Root**:
- Static snapshots don't detect live tampering
- Need continuous verification of critical files (/etc/shadow, kernel modules)
- DAG records historical state but not real-time drift

**Recommendation**:
- Implement inotify/FSEvents watchers for critical paths
- Generate DAG nodes on unauthorized file modifications
- Link FIM events to compliance controls (STIG requirement violations)

**Priority**: HIGH (foundational for continuous attestation)

---

### Gap 2: Network Trust Flow Modeling
**Current State**: Sonar scans network but doesn't model lateral movement paths

**Why This is a Missing Root**:
- Current: Isolated findings (Port 22 open on Host A, RDP on Host B)
- Needed: "If Host A compromised, attacker can pivot to Host B via SMB"
- Blast radius analysis requires trust flow graphs

**Recommendation**:
- Build network topology DAG (assets + connections)
- Model privilege escalation paths (service accounts, shared credentials)
- Generate "attack graph" overlays on trust constellation

**Priority**: MEDIUM (enhances causal modeling depth)

---

### Gap 3: SBOM (Software Bill of Materials) Integration
**Current State**: Clean-room IP verification exists but no active SBOM tracking

**Why This is a Missing Root**:
- Supply chain attacks (SolarWinds, Log4Shell) require dependency awareness
- Current CVE correlation limited to known vulnerabilities
- Need proactive component tracking (what libraries are deployed?)

**Recommendation**:
- Integrate SBOM generation (SPDX/CycloneDX formats)
- Auto-correlate SBOM components with CVE database
- Generate DAG nodes: `component:library:log4j` → `vuln:CVE-2021-44228`

**Priority**: MEDIUM (addresses supply chain root causes)

---

### Gap 4: Zero-Trust Network Access (ZTNA) Enforcement
**Current State**: Tailscale mesh mentioned but not enforced

**Why This is a Missing Root**:
- Current: Optional mTLS for agent communication
- Needed: Mandatory identity verification + policy enforcement before access
- Aligns with "never trust, always verify" principle

**Recommendation**:
- Integrate with Tailscale ACLs or native eBPF network policies
- Enforce DAG-verified device posture before network access
- Link access decisions to real-time attestation state

**Priority**: LOW (operational enhancement, not core attestation)

---

### Gap 5: Threat Hunting Query Language
**Current State**: Sonar collects data; humans/LLM interpret

**Why This is a Missing Root**:
- Need structured queries against DAG (e.g., "Find all hosts with SSH exposed AND outdated OpenSSL")
- Current: Manual correlation of findings
- Needed: Declarative threat hunting (like Splunk SPL or KQL)

**Recommendation**:
- Implement DAG query DSL: `FIND asset:host WHERE port:22 OPEN AND cve:OpenSSL CRITICAL`
- Enable automated threat hunting workflows
- Generate recurring attestations for specific threat patterns

**Priority**: MEDIUM (amplifies intelligence capabilities)

## 4. Recommended Enhancements (Deepening the Roots)

### Enhancement 1: Adversarial Simulation (Red Team Automation)
**Objective**: Move from passive scanning to active adversarial testing

**Implementation**:
- Integrate Atomic Red Team (MITRE ATT&CK technique emulation)
- Generate DAG nodes for simulated attacks vs actual defenses
- Validate STIG controls under adversarial pressure

**Impact**: Proves controls work in practice, not just on paper

---

### Enhancement 2: Causal Risk Scoring (Beyond CVSS)
**Objective**: Replace CVSS with context-aware risk scores

**Implementation**:
- Factor: Public exposure (Shodan visibility) + exploit availability + privilege level
- Example: CVE-2023-1234 (CVSS 7.5) escalates to 9.5 if publicly exposed on internet
- Link risk scores to business impact (service downtime, data exfiltration)

**Impact**: Prioritizes remediation based on actual threat landscape

---

### Enhancement 3: Automated Remediation Workflows
**Objective**: Close the loop from attestation to enforcement

**Implementation**:
- Generate Ansible/Terraform remediation playbooks from DAG findings
- Example: "Port 22 exposed" → Auto-apply firewall rule to restrict SSH
- Require human approval for destructive actions (policy gates)

**Impact**: Moves from "detection" to "correction" (operational security)

---

### Enhancement 4: Provenance Tracking for AI-Generated Code
**Objective**: Address "GenAI crash" risk (scaling leaves, skipping roots)

**Implementation**:
- Tag AI-generated code in SBOM with model ID + prompt hash
- Track lineage: "This function was generated by GPT-4 on 2025-12-20"
- Flag unvetted AI code for human review before production deployment

**Impact**: Prevents IP contamination and ensures code quality
